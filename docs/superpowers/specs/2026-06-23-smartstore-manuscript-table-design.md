# 스마트스토어 표 입력용 원고 개선 설계

## 배경

Truck Harvester는 truck-no1 매물 주소에서 차량 이미지, 성능점검기록부,
원고 파일을 차량별 폴더로 저장한다. 사용자는 저장된 원고와 이미지를
바탕으로 네이버 스마트스토어 상품 상세 페이지를 사람이 직접 등록한다.

실제 스마트스토어 상세 페이지에는 상단에 차량 요약 표가 들어간다. 이 표는
`차명`, `연식`, `주행거리`, `차량번호`, `차량정보` 행으로 구성된다. 현재
원고 파일은 `기타사항`에 truck-no1의 `추가장착 옵션` 값을 넣기 때문에,
상세설명 안의 `상부`, `하부` 정보를 보고 표를 채우기 어렵다.

예시 매물
`https://www.truck-no1.co.kr/model/DetailView.asp?ShopNo=30195108&MemberNo=1000294965&OnCarNo=2025300457962`
에서는 사람이 등록한 스마트스토어 표의 `차량정보`가 `상부`, `하부`로
정리되어 있다. 그러나 현재 원고는 같은 정보를 구조화하지 못하고
`추가장착 옵션` 나열만 저장한다.

## 목표

- 원고 파일의 `기타사항`만 보고 스마트스토어 표 5개 행을 채울 수 있게 한다.
- `연식`은 `년형 | 등록` 행의 `최초등록` 날짜를 기준으로
  `yyyy년 m월 등록` 형식으로 표기한다.
- 상세설명 안의 `차명`, `상부`, `하부` 값이 비어 있어도 원고가 이상하게
  보이지 않도록 fallback을 명확히 한다.
- 기존 원고 파일 이름과 저장 폴더 구조는 바꾸지 않는다.
- parser, entity contract, 원고 생성 책임을 분리해 이후 테스트와 확장을
  쉽게 한다.

## 비목표

- 네이버 스마트스토어 업로드 자동화는 이번 범위에 포함하지 않는다.
- 원고 전체 템플릿과 영업 문구를 다시 디자인하지 않는다.
- 차량 이미지, 성능점검기록부 저장 방식은 바꾸지 않는다.
- truck-no1 외 다른 매물 사이트의 상세설명 구조는 지원하지 않는다.
- `추가장착 옵션`을 사용자 원고의 표 입력 fallback으로 노출하지 않는다.

## 선택한 접근

`TruckListing`에 스마트스토어 표 입력용 구조화 field를 추가한다.

기존 `options` field는 유지한다. `options`는 truck-no1의
`추가장착 옵션`에서 온 값이고, 이번에 필요한 값은 스마트스토어 표에 맞춘
별도 의미를 갖는다. 두 의미를 같은 field에 섞으면 이름과 책임이 어긋난다.

새 구조화 field를 parser에서 만들고, 원고 생성기는 이 field만 읽어
`기타사항` 블록을 출력한다. 이렇게 하면 scraping 의미는
`src/v2/shared/lib/parse-truck-html.ts`에 남고, 파일 저장 feature는 HTML 구조를
알 필요가 없다.

## 데이터 계약

`truckListingSchema`에 `smartStoreTable`을 추가한다.

```ts
smartStoreTable: {
  vehicleName: string
  registrationLabel: string
  mileage: string
  vehicleNumber: string
  upperInfo: string
  lowerInfo: string
  hasVehicleInfo: boolean
}
```

각 field의 의미는 다음과 같다.

- `vehicleName`: 스마트스토어 표의 `차명` 행에 넣을 값.
- `registrationLabel`: 스마트스토어 표의 `연식` 행에 넣을 값.
- `mileage`: 스마트스토어 표의 `주행거리` 행에 넣을 값.
- `vehicleNumber`: 스마트스토어 표의 `차량번호` 행에 넣을 값.
- `upperInfo`: `차량정보` 아래 `상부` 값. 값이 없으면 `정보 없음`.
- `lowerInfo`: `차량정보` 아래 `하부` 값. 값이 없으면 `정보 없음`.
- `hasVehicleInfo`: `상부`나 `하부` 중 하나라도 원본 상세설명에서 실제 값을
  찾았는지 나타낸다.

`upperInfo`와 `lowerInfo`는 항상 문자열을 갖는다. 둘 중 하나만 있어도 원고는
`상부`, `하부`를 모두 출력하고, 빠진 쪽은 `정보 없음`으로 표기한다.
둘 다 없으면 원고는 `차량정보 : 정보 없음` 한 줄로 출력한다.

## Parser 규칙

### 연식

`car-detail`의 `dt` 텍스트가 `년형 | 등록`인 행을 찾고, 바로 다음 `dd`에서
`최초등록` 텍스트와 함께 있는 8자리 날짜를 추출한다.

```html
<dt>년형 | 등록</dt>
<dd><strong class="number">2023</strong>년형 | <span class="number">20231117</span> 최초등록</dd>
```

위 값은 `2023년 11월 등록`으로 변환한다. 월 앞의 `0`은 제거한다.
날짜가 없거나 8자리 날짜가 유효하지 않으면 기존 `year` 값을 사용한다.

### 차명

`▶ 상세설명 ::` 이후 문단에서 `차명:` label을 찾는다. label 뒤에 실제 값이
있으면 그 값을 `smartStoreTable.vehicleName`으로 사용한다.

`차명:`만 있고 값이 비어 있으면 기존 `vehicleName` fallback을 사용한다.
빈 label 자체를 원고에 내보내지 않는다.

### 차량정보

`▶ 상세설명 ::` 이후 문단에서 `상부:`와 `하부:` label을 찾는다.
각 label 뒤에 실제 값이 있으면 값을 보존한다. `·`, `&nbsp;`, 중복 공백,
스타일용 빈 문단, `<br>`만 있는 문단은 제거한다. 괄호, 쉼표, 한글 설명은
보존한다.

`상부:` 또는 `하부:` 뒤의 설명이 다음 문단으로 이어질 수 있다. 이 경우
다음 `차명:`/`상부:`/`하부:` label 또는 판매자 소개 구분선이 나오기
전까지의 문단을 같은 field 값으로 보존한다. 예를 들어 `상부:` 첫 줄 뒤에
점검 완료 문구와 작업 높이/반경 문구가 별도 문단으로 이어지면 세 문단을
모두 `upperInfo`에 저장한다.

`상부:`와 `하부:`가 label만 있고 값이 없을 수 있다. 예시 매물
`https://www.truck-no1.co.kr/model/DetailView.asp?ShopNo=30195108&MemberNo=1000294965&OnCarNo=2024300329293`
가 이 케이스다.

규칙은 다음과 같다.

- `상부`와 `하부` 모두 값이 있으면 둘 다 출력한다.
- 둘 중 하나만 값이 있으면 둘 다 출력하고, 빠진 쪽은 `정보 없음`으로 출력한다.
- 둘 다 값이 없으면 `차량정보 : 정보 없음` 한 줄로 출력한다.
- `추가장착 옵션`은 `차량정보` fallback 문구로 노출하지 않는다.

### 주행거리와 차량번호

`주행거리`와 `차량번호`는 기존 parser가 추출한 `mileage`, `vnumber` 값을
사용한다. 기존 fallback 문구가 들어온 경우에도 원고에 그대로 남겨 등록자가
확인할 수 있게 한다.

## 원고 출력

원고 파일 이름은 바꾸지 않는다. 기존 `기타사항` 블록의 내용만 바꾼다.

값이 충분한 매물은 다음처럼 출력한다.

```text
기타사항 :
  차명 : 포터2 1.3톤활어운반차
  연식 : 2023년 11월 등록
  주행거리 : 159,600km
  차량번호 : 822수2698
  차량정보 :
    상부 : 인증차(바로1.3톤활어운반차), 액산병 보유
    하부 : 1인신조, 무사고, 오토미션
```

`상부`나 `하부` 값이 여러 줄이면 첫 줄에는 label을 붙이고, 이어지는 줄은
같은 하위 항목임을 알 수 있도록 한 단계 더 들여쓴다.

```text
기타사항 :
  차명 : 봉고3 1톤930바가지차
  연식 : 2023년 2월 등록
  주행거리 : 91,000km
  차량번호 : 90로1234
  차량정보 :
    상부 : 동해930, 인버터,유/무선리모컨, 작업다이, 공구함
      동해기계항공 용인 서비스센터에서 점검 완료
      작업 높이: 8.9m, 작업 반경: 6.4m
    하부 : 오토미션, 133마력, 요소수 타입, 실내클리닝 완료
```

`상부`나 `하부` 중 하나만 있는 매물은 두 줄을 모두 유지한다.

```text
기타사항 :
  차명 : 포터2 특장차
  연식 : 2023년 11월 등록
  주행거리 : 159,600km
  차량번호 : 822수2698
  차량정보 :
    상부 : 인증차(바로1.3톤활어운반차), 액산병 보유
    하부 : 정보 없음
```

상세설명에 `상부`, `하부` 값이 모두 비어 있는 매물은 다음처럼 출력한다.

```text
기타사항 :
  차명 : 냉동탑 5톤이상
  연식 : 2021년 8월 등록
  주행거리 : 363,313km
  차량번호 : 91누5384
  차량정보 : 정보 없음
```

들여쓰기는 `기타사항` 아래 항목이 한 묶음임을 보여주기 위한 것이다.
`상부`, `하부`는 `차량정보`의 하위 항목이므로 한 단계 더 들여쓴다.

## 에러 처리와 fallback

- `최초등록` 날짜가 없으면 기존 `year` 값을 사용한다.
- 상세설명 `차명:` 값이 없으면 기존 `vehicleName` 값을 사용한다.
- `상부`, `하부` 값이 모두 없으면 `차량정보 : 정보 없음`으로 출력한다.
- `상부`, `하부` 중 하나만 없으면 빠진 쪽만 `정보 없음`으로 출력한다.
- parser가 예상하지 못한 상세설명 HTML을 만나도 매물 parsing 전체를 실패로
  바꾸지 않는다.
- 원고에는 빈 `차명 :`, 빈 `상부 :`, 빈 `하부 :` 줄을 남기지 않는다.

## 테스트 계획

- entity schema test
  - `smartStoreTable` field를 가진 `TruckListing`을 허용한다.
  - `upperInfo`, `lowerInfo`, `hasVehicleInfo`가 기대 타입으로 검증된다.
- parser unit test
  - `최초등록` 날짜 `20231117`을 `2023년 11월 등록`으로 변환한다.
  - `상세설명`에서 `차명`, `상부`, `하부` 값을 추출한다.
  - `상부` 또는 `하부` 값이 label 이후 여러 문단으로 이어지면 모두 보존한다.
  - `차명:`, `상부:`, `하부:`가 label만 있을 때 빈 값을 사용하지 않는다.
  - `상부` 또는 `하부` 한쪽만 있을 때 빠진 쪽을 `정보 없음`으로 채운다.
  - `성능점검보기` URL 추출 등 기존 parser 동작은 유지된다.
- file-system manuscript test
  - 저장되는 원고에 들여쓰기된 스마트스토어 표용 `기타사항` 블록이 포함된다.
  - 여러 줄 `상부`/`하부` 값의 후속 줄이 하위 항목으로 들여쓰기된다.
  - 빈 상세설명 케이스에서 `차량정보 : 정보 없음`이 포함된다.
  - 원고에 `추가장착 옵션 : 정보 없음`이 나타나지 않는다.
- ZIP fallback test
  - ZIP 안의 원고 파일에도 같은 `기타사항` 블록이 저장된다.

## 구현 경계

HTML 해석과 상세설명 fallback은 `src/v2/shared/lib/parse-truck-html.ts`가
담당한다. `TruckListing` 구조 변경은 `src/v2/entities/truck`에서 검증한다.
원고 텍스트 조립은 `src/v2/features/file-management/text-content.ts`에서
`smartStoreTable`을 읽어 처리한다.

file-management layer는 truck-no1 HTML selector를 알지 않는다. 이 경계를
유지하면 저장 방식이 바뀌어도 scraping 규칙과 원고 출력 규칙을 독립적으로
테스트할 수 있다.
