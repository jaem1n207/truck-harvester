# Umami 작업 퍼널 분석 설계

작성일: 2026-04-29

## 배경

Truck Harvester의 핵심 사용 흐름은 `붙여넣기 -> 매물 확인 -> 저장 폴더 또는 ZIP 저장 -> 완료 확인`이다. 방문자 수보다 중요한 것은 직원이 실제 업무를 끝냈는지, 어느 단계에서 멈췄는지, 실패가 어떤 매물과 단계에서 반복되는지다.

이 설계는 Umami Cloud 무료 플랜을 사용해 제품 분석 데이터를 수집한다. 사용자는 분석 우선순위를 다음 순서로 확정했다.

1. 저장 성공까지 이어지는 작업 전환율
2. 실제 사용량, 배치 크기, 작업 빈도
3. 파싱/저장 실패 원인과 복구 가능성
4. 온보딩/도움말 효과

성공 전환 기준은 `확인 성공한 매물이 모두 저장 완료됨`이다. 붙여넣은 원문에 잘못된 주소가 섞여도, 확인 가능한 매물이 모두 저장됐다면 실무상 성공으로 본다.

## 목표

- 작업 단위 퍼널에서 저장 완료율과 이탈 지점을 볼 수 있게 한다.
- 10주소 배치가 실제로 자주 쓰이는지, 평균 배치 크기와 처리 시간을 볼 수 있게 한다.
- 확인 실패, 무효 주소, 저장 실패가 어떤 비율로 발생하는지 분류한다.
- 실패 매물에 한해 실제 URL, 차량번호, 차명을 수집해 반복 실패를 추적할 수 있게 한다.
- 성공 매물의 URL, 차량번호, 차명은 수집하지 않는다.
- Umami가 없거나 차단되어도 앱 기능이 깨지지 않게 한다.

## 확정 접근

접근은 `작업 퍼널 + 실패 매물 식별 이벤트`로 확정한다.

작업 단위 이벤트는 모든 주요 단계를 기록한다. 배치 이벤트에는 개수, 시간, 저장 방식, 브라우저 기능 지원 여부만 담는다. 매물별 상세 이벤트는 실패 건에만 기록한다.

성공 매물은 배치 요약에 개수로만 포함한다. `listing_url`, `vehicle_number`, `vehicle_name`은 실패 매물 이벤트에만 포함한다. 파싱 실패나 무효 주소처럼 차량 정보가 없는 경우에는 실제 URL 또는 제한된 입력 샘플과 실패 분류만 보낸다.

사용자가 취소한 작업과 중복 주소는 실패율에 포함하지 않는다. 이 신호는 입력 품질 또는 작업 중단 분석에만 사용한다.

## 이벤트 목록

### `batch_started`

붙여넣기에서 처리 가능한 후보 주소가 생긴 순간 기록한다.

용도:

- 작업 시작 수 집계
- 붙여넣기당 후보 주소 수 집계
- 중복 제거 전후 입력 품질 확인

### `preview_completed`

매물 확인 작업이 끝난 순간 기록한다.

용도:

- 매물 확인 단계 성공률 확인
- 확인 실패와 무효 주소 비율 확인
- 저장 전 이탈 지점 계산

### `save_started`

사용자가 저장 버튼을 눌러 저장을 시작한 순간 기록한다.

용도:

- 확인 완료 뒤 저장을 시작하지 않은 작업 비율 계산
- 저장 방식별 사용량 확인

### `save_completed`

확인 성공한 매물이 모두 저장된 순간 기록한다.

용도:

- 핵심 성공 전환 계산
- 평균 작업 완료 시간 계산
- 폴더 저장과 ZIP 저장의 성공률 비교

### `save_failed`

저장 과정에서 하나 이상 실패한 순간 기록한다.

용도:

- 저장 단계 실패율 계산
- 저장 방식별 실패율 비교
- 저장 실패 뒤 복구 필요 작업 확인

### `listing_failed`

확인 실패, 무효 주소, 지원하지 않는 입력, 저장 실패가 발생한 매물 단위
상세 이벤트다. 지원하지 않는 입력은 매물 후보를 하나도 추출하지 못한
비어 있지 않은 붙여넣기를 뜻하며, 붙여넣기 1회당 최대 1건만 전송한다.

용도:

- 실패 유형별 Top N 확인
- 특정 URL, 차량번호, 차명에서 반복되는 실패 패턴 확인
- 파싱 실패와 저장 실패의 원인 분리

## 배치 이벤트 속성

모든 배치 이벤트는 가능한 범위에서 다음 속성을 공유한다.

- `batch_id`: 한 번의 붙여넣기/저장 흐름을 묶는 임의 ID
- `url_count`: 붙여넣기에서 추출된 후보 주소 수
- `unique_url_count`: 중복 제거 후 처리한 주소 수
- `ready_count`: 확인 성공한 매물 수
- `invalid_count`: 주소는 있었지만 매물로 쓰기 어려운 수
- `preview_failed_count`: 매물 확인 실패 수
- `saved_count`: 저장 완료 수
- `save_failed_count`: 저장 실패 수
- `duration_ms`: 해당 단계까지 걸린 시간
- `save_method`: `directory` 또는 `zip`
- `filesystem_supported`: 폴더 저장 가능 여부
- `notification_enabled`: 완료 알림 활성 여부

`batch_id`는 사용자 식별자가 아니다. 브라우저 세션이나 사용자 계정에 묶지 않는 임의 작업 ID로 만든다.

## 실패 매물 이벤트 속성

`listing_failed`에는 다음 속성을 담는다.

- `batch_id`
- `failure_stage`: `invalid_url`, `preview`, `save`
- `failure_reason`: 사람이 읽기 쉬운 짧은 분류값
- `listing_url`: 실패한 실제 URL. `failure_stage=invalid_url`이고
  `failure_reason=unsupported_input`인 경우에는 사용자가 붙여넣은 입력 샘플일
  수 있다.
- `vehicle_number`: 이미 확인된 매물의 저장 실패인 경우만
- `vehicle_name`: 이미 확인된 매물의 저장 실패인 경우만
- `image_count`: 이미 확인된 경우만
- `elapsed_ms`: 해당 매물 처리에 걸린 시간
- `input_was_truncated`: 지원하지 않는 입력 샘플이 160자를 넘어 잘렸는지 여부

확인 실패나 무효 주소는 차량 정보를 알 수 없을 수 있으므로 `vehicle_number`, `vehicle_name`, `image_count`를 보내지 않는다.

지원하지 않는 입력 샘플은 앞뒤 공백을 제거하고 연속 공백을 한 칸으로 축약한 뒤
160자까지만 저장한다. 공백뿐인 입력은 실패 이벤트로 보내지 않는다.

## 분석 질문

주간/월간으로 다음 질문에 답할 수 있어야 한다.

- 전체 작업 중 몇 퍼센트가 `save_completed`까지 갔는가?
- `preview_completed`까지는 갔지만 저장을 시작하지 않은 작업은 얼마나 되는가?
- 평균 배치 크기는 몇 개인가?
- 10주소 안팎의 배치가 실제로 얼마나 자주 쓰이는가?
- 확인 성공 매물이 1대 이상 있었는데 저장 실패로 끝난 작업은 얼마나 되는가?
- 폴더 저장과 ZIP fallback 중 어느 쪽에서 실패율이 높은가?
- 실패는 `invalid_url`, `preview`, `save` 중 어디에 몰리는가?
- 실패한 실제 URL, 차량번호, 차명 기준으로 반복 실패 패턴이 있는가?
- 알림을 켠 사용자와 끈 사용자 사이에 저장 완료율 차이가 있는가?

Umami 기본 방문, 디바이스, 브라우저 지표는 보조 지표로만 쓴다. 예를 들어 특정 브라우저에서 `filesystem_supported=false`와 ZIP 사용률이 높다면 fallback 문구나 저장 안내를 개선할 근거가 된다.

## 구현 경계

앱 로직 곳곳에서 `umami.track()`을 직접 호출하지 않는다. 제품 언어에 맞춘 얇은 분석 래퍼를 둔다.

예상 공개 함수:

- `trackBatchStarted`
- `trackPreviewCompleted`
- `trackSaveStarted`
- `trackSaveCompleted`
- `trackSaveFailed`
- `trackListingFailed`
- `trackUnsupportedInputFailure`

이 래퍼는 다음 책임을 가진다.

- `window.umami` 존재 여부 확인
- Umami 호출 실패 삼키기
- 이벤트 이름 안정화
- 성공 매물 식별 정보 제거
- 실패 매물 식별 정보만 허용
- 숫자, 문자열, 불리언 형태의 이벤트 속성 정규화

## 운영 주의사항

- Umami Cloud에 실패 매물의 실제 URL, 차량번호, 차명이 저장된다.
- 이 식별 정보 수집은 사용자가 명시적으로 승인한 범위다.
- 성공 매물의 실제 URL, 차량번호, 차명은 수집하지 않는다.
- 사용자가 취소한 작업은 실패율에 포함하지 않는다.
- 중복 주소는 실패가 아니라 입력 품질 신호로만 본다.
- 이미지별 저장 이벤트는 만들지 않는다.
- 이벤트 이름은 짧고 안정적으로 유지한다.
- 무료 플랜 사용량을 고려해 배치 이벤트와 실패 상세 이벤트만 남긴다.
- 지원하지 않는 입력은 무료 플랜 사용량을 보호하기 위해 붙여넣기 1회당 최대
  `listing_failed` 1건만 남긴다.

## 테스트 계획

- 이벤트 payload 생성 단위 테스트를 추가한다.
- 성공 매물 payload에 `listing_url`, `vehicle_number`, `vehicle_name`이 들어가지 않는지 검증한다.
- 실패 매물 payload에는 실패 단계에 맞는 식별 정보가 들어가는지 검증한다.
- `window.umami`가 없어도 track 함수가 예외를 던지지 않는지 검증한다.
- `window.umami.track`이 예외를 던져도 사용자 작업이 계속되는지 검증한다.
- 붙여넣기, 매물 확인 완료, 저장 시작, 저장 완료, 저장 실패 흐름에서 올바른 track 함수가 호출되는지 검증한다.
- 취소와 중복 주소가 실패 이벤트로 기록되지 않는지 검증한다.

## 범위 밖

- Umami 대시보드 커스텀 구현은 이번 범위에서 제외한다.
- 자가 호스팅 Umami 구축은 제외한다.
- 사용자 계정, 직원명, 전화번호 같은 사용자 식별자 수집은 제외한다.
- 성공 매물의 실제 URL, 차량번호, 차명 수집은 제외한다.
- 이미지별 다운로드 성공/실패 이벤트 수집은 제외한다.
- 외부 에러 모니터링 SDK 추가는 제외한다.
