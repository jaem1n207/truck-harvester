# 지원하지 않는 입력 분석 설계

작성일: 2026-04-30

## 배경

Truck Harvester는 붙여넣은 내용에서 지원하는 트럭 매물 상세 주소를
추출한 뒤 매물 확인을 시작한다. 현재는 `DetailView.asp?...`처럼 앞부분이
빠진 주소, `abc...` 같은 임의 텍스트, 지원하지 않는 사이트 주소를 붙여넣으면
화면에 `매물 주소를 찾지 못했어요. 복사한 내용을 다시 확인해 주세요.`만
보이고 Umami 이벤트는 남지 않는다.

사용자는 이 실패도 사실상 "주소 입력이 잘못되어 매물 처리가 실패한 것"으로
보고 싶어 한다. 데이터가 쌓이면 사용자가 입력란에 어떤 값을 허용된 값으로
인지하는지, 어떤 입력 패턴 때문에 매물 처리가 시작되지 않는지 검증할 수 있다.

이 설계는 기존 `docs/superpowers/specs/2026-04-29-umami-analytics-design.md`
의 `listing_failed` 이벤트 모델을 확장한다. 새 이벤트 종류를 만들지 않고,
지원하지 않는 입력도 `listing_failed`의 `invalid_url` 실패로 통합한다.

## 목표

- 지원하는 매물 주소를 하나도 추출하지 못한 비어 있지 않은 붙여넣기를
  Umami에 남긴다.
- 기존 `listing_failed` 이벤트로 통합해 실패 분석 축을 하나로 유지한다.
- Umami Hobby 플랜의 월 100K 이벤트 제한에 부담을 크게 주지 않는다.
- 비즈니스 파싱 로직, UI 컴포넌트, 분석 추적 로직의 책임을 분리한다.
- 사용자가 보는 UI 문구와 입력 흐름은 그대로 유지한다.

## 비목표

- 잘못된 입력 조각마다 이벤트를 여러 건 전송하지 않는다.
- 정상 매물 주소가 하나라도 추출된 붙여넣기의 주변 일반 텍스트를 실패로
  기록하지 않는다.
- 새 Umami 대시보드, 자가 호스팅, 사용자 식별자 수집은 만들지 않는다.
- 성공 매물의 실제 URL, 차량번호, 차명 수집 정책은 변경하지 않는다.

## 확정 접근

접근은 "실패한 붙여넣기당 `listing_failed` 최대 1건"으로 확정한다.

`parseUrlInputText(text)`가 실패를 반환했고, 원문이 공백만 있는 값이 아니라면
분석 계층에서 `listing_failed`를 한 번 보낸다.

이벤트 속성은 다음처럼 만든다.

- `failure_stage`: `invalid_url`
- `failure_reason`: `unsupported_input`
- `listing_url`: 사용자가 붙여넣은 원문을 정리한 짧은 입력 샘플
- `elapsed_ms`: 이 실패 입력을 처리하는 데 걸린 시간
- `batch_id`: 해당 실패 입력을 묶는 임의 작업 ID
- `input_was_truncated`: 입력 샘플이 최대 길이 때문에 잘렸는지 여부

`listing_url` 이름은 기존 실패 이벤트 계약을 유지하기 위해 그대로 둔다.
이 값은 `invalid_url` 단계에서는 실제 URL이 아니라 "사용자가 주소로 넣은 입력
샘플"일 수 있다.

## 입력 샘플 제한

무료 플랜 이벤트 사용량과 이벤트 데이터 크기를 함께 아끼기 위해 입력 샘플은
보수적으로 줄인다.

정리 규칙:

1. 앞뒤 공백을 제거한다.
2. 줄바꿈, 탭, 연속 공백을 한 칸으로 축약한다.
3. 최대 160자까지만 전송한다.
4. 정리 후 빈 문자열이면 이벤트를 보내지 않는다.

160자는 `DetailView.asp?ShopNo=...&MemberNo=...&OnCarNo=...`처럼 앞부분이
빠진 핵심 실패 패턴을 대부분 담을 수 있고, 긴 채팅 본문이 그대로 저장되는
위험은 낮춘다. 데이터 품질이 부족하다고 확인되면 이 상수만 조정할 수 있게
분석 계층에 모아 둔다.

## 책임 분리

비즈니스 로직과 추적 로직은 섞지 않는다.

- `parseUrlInputText`: 입력에서 지원하는 매물 주소를 추출하고, 성공/실패와
  사용자 안내 문구만 반환한다. Umami, 이벤트명, 길이 제한 규칙을 모른다.
- `ListingChipInput`: 붙여넣은 텍스트를 상위로 전달하고 화면 상태를 렌더링한다.
  분석 이벤트를 직접 호출하지 않는다.
- `analytics.ts`: 이벤트명, payload 변환, 입력 샘플 정리, `window.umami` 안전
  호출을 담당한다.
- root route composition 또는 전용 custom hook: 파싱 결과를 보고 분석 계층의
  함수를 호출한다. payload 세부 필드는 직접 조립하지 않는다.

분석 계층에는 `trackUnsupportedInputFailure` 같은 전용 함수를 둔다. 호출부는
`rawInput`과 처리 시간에 필요한 최소 정보만 넘기고, 실제 `listing_failed`
payload 구성은 분석 계층에서 처리한다.

## 데이터 흐름

1. 사용자가 입력란에 텍스트를 붙여넣는다.
2. `ListingChipInput`이 원문을 `onPasteText(text)`로 넘긴다.
3. route composition이 `parseUrlInputText(text)`를 호출한다.
4. 파싱 실패이고 정리한 원문이 비어 있지 않으면 분석 계층에 실패 입력을 알린다.
5. 분석 계층이 `listing_failed` 이벤트를 최대 1건 전송한다.
6. UI는 현재와 동일하게 복구 안내 문구를 보여준다.

후보 URL이 하나 이상 추출되면 현재 preview 흐름을 그대로 사용한다. 실제로
지원 URL을 요청했지만 사이트 응답 실패, 누락된 매물 정보, 차명 정보 없음으로
실패한 경우도 기존 `listing_failed` 경로를 그대로 사용한다.

## 이벤트 사용량 방어

- 비어 있는 입력은 이벤트를 보내지 않는다.
- 후보 URL이 0개인 붙여넣기에서만 unsupported input 이벤트를 보낸다.
- 붙여넣기 1회당 이벤트는 최대 1건이다.
- 잘못된 조각을 여러 개 추출하거나 여러 건으로 나누지 않는다.
- `batch_started`와 `preview_completed`는 후보 URL이 있는 작업에만 유지한다.
  지원하지 않는 입력은 `listing_failed` 1건만 남긴다.

이 방식은 지원하지 않는 입력을 관측하면서도, 사용자가 긴 본문을 반복해서
붙여넣는 경우 이벤트 수가 입력 횟수와 같은 속도로만 증가하도록 제한한다.

## 오류 처리

분석 전송 실패는 사용자 작업을 방해하지 않는다. `window.umami`가 없거나,
광고 차단 등으로 tracker가 동작하지 않거나, `track()` 호출이 예외를 던져도
입력 안내와 매물 저장 흐름은 계속되어야 한다.

사용자에게 보이는 문구는 바꾸지 않는다. 지원하지 않는 입력의 UI 결과는 기존과
동일하게 `매물 주소를 찾지 못했어요. 복사한 내용을 다시 확인해 주세요.`다.

## 문서 갱신

구현 시 기존 Umami 설계 문서에도 다음 결정을 반영한다.

- 지원하지 않는 입력은 `listing_failed / invalid_url`로 통합한다.
- 붙여넣기 1회당 최대 1건만 전송한다.
- `listing_url`은 `invalid_url` 단계에서 입력 샘플일 수 있다.
- 입력 샘플은 160자로 제한한다.
