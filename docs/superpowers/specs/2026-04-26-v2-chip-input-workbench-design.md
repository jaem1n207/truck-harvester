# v2 칩 입력 작업판 UX 설계

작성일: 2026-04-26

## 배경

Truck Harvester v2의 주 사용자는 인터넷 사용에 익숙하지 않은 40대 중고차 딜러와 막 직장 생활을 시작한 비개발자 직원이다. 현재 화면은 URL 입력, 저장 폴더, 가져올 매물 목록, 진행 상황이 분리되어 있어 사용자가 다음 행동을 헷갈릴 수 있다. 특히 URL 원문과 `truck-1` 같은 내부 식별자는 사람에게 의미가 약하고, 목록의 `x` 버튼이 실제 진행 취소처럼 보이는 문제가 있다.

이 설계는 기존 `/` route와 legacy 동작을 유지하면서 `/v2` 범위에서만 적용한다. v2에는 Sentry와 watermark 기능을 추가하지 않는다.

## 목표

- 사용자가 줄바꿈이나 URL 형식을 기억하지 않아도 매물 주소를 넣을 수 있게 한다.
- 붙여넣은 주소를 즉시 사람이 읽기 쉬운 매물명 칩으로 바꾼다.
- 시작 전 제거와 시작 후 진행 상태를 명확히 분리한다.
- 저장 완료를 모달 없이 화면 안에서 분명하게 알린다.
- 데스크톱 알림은 선택형 보조 기능으로 둔다.

## 참고 원칙

- NN/g의 사용성 휴리스틱 중 상태 가시성, 현실 언어 사용, 오류 예방, 기억보다 인지, 오류 복구 원칙을 따른다. 참고: https://www.nngroup.com/articles/ten-usability-heuristics/
- 오류 문구는 무엇이 잘못됐고 어떻게 고치는지 짧고 쉬운 말로 설명한다. GOV.UK Design System의 error message guidance를 참고한다. 참고: https://design-system.service.gov.uk/components/error-message/
- 상태 배지는 짧고 스캔 가능한 말과 의미 있는 색으로 표현한다. Shopify Polaris Badge와 Microsoft Fluent Badge guidance를 참고한다. 참고: https://polaris.shopify.com/components/feedback-indicators/badge, https://fluent2.microsoft.design/components/web/react/core/badge/usage
- Notion multi-select처럼 여러 항목을 입력칸 안의 칩으로 다루는 패턴을 참고하되, 트럭 매물 업무에 맞게 매물명과 저장 상태를 우선한다. 참고: https://www.notion.com/help/database-properties

## 확정 접근

접근은 `입력칸 중심 자동 확인형`으로 확정한다.

사용자는 한 화면에서 복사한 내용을 그대로 붙여넣는다. 앱은 텍스트 안에서 지원하는 매물 URL만 자동으로 골라내고, URL을 즉시 `확인 중` 칩으로 바꾼다. 매물 정보 확인에 성공하면 칩 레이블을 `[활어차] 포터2 슈퍼캡/초장축/(CRDi)` 같은 매물명으로 바꾼다.

별도의 `가져올 매물` URL 목록은 사용자에게 노출하지 않는다. 시작 전 준비 목록은 칩 입력칸이 담당하고, 시작 후 진행 상황은 매물명 중심 진행 목록이 담당한다.

## 사용자 흐름

1. 사용자가 카톡, 문자, 브라우저에서 복사한 내용을 `매물 주소 넣기` 칸에 붙여넣는다.
2. 앱이 붙여넣은 내용에서 지원하는 매물 URL만 자동 추출한다.
3. 새 URL은 입력칸 안에서 `확인 중 · 매물 이름 찾는 중` 칩으로 표시된다.
4. 매물 정보 확인에 성공하면 칩은 실제 매물명으로 바뀐다.
5. 지원하지 않는 주소이거나 확인에 실패하면 `주소 확인 필요` 칩으로 바뀐다.
6. 시작 전에는 칩의 `x` 또는 `지우기`로 제거할 수 있다.
7. 실패 칩은 직접 URL을 수정하지 않는다. 복구 방식은 `지우기 -> 다시 붙여넣기`로 한다.
8. `확인된 N대 저장 시작`을 누르면 확인된 매물만 저장한다.
9. 저장 폴더가 없으면 한 번만 폴더 선택 화면을 띄우고, 선택한 폴더를 현재 저장 폴더로 표시한다.
10. 저장 중에는 제거 버튼을 숨기거나 비활성화한다.
11. 완료되면 진행 상황 영역에서 `N대 저장 완료`를 크게 보여준다.
12. 사용자가 원하면 `완료 알림 켜기`로 데스크톱 알림을 켤 수 있다.

## 화면 구성

화면은 `한 화면 작업판`으로 구성한다.

### 왼쪽: 매물 주소 넣기

- 제목: `매물 주소 넣기`
- 안내 문구: `복사한 내용을 그대로 붙여넣으세요. 매물 주소만 자동으로 찾습니다.`
- 입력 영역은 textarea처럼 보이지만 내부에 칩이 쌓이는 tag input 형태다.
- 사용자는 여러 줄, 띄어쓰기, 문장이 섞인 내용을 붙여넣을 수 있다.
- 칩 상태:
  - `checking`: `매물 이름 찾는 중`
  - `ready`: 매물명
  - `invalid`: `주소 확인 필요`
  - `failed`: `매물 이름을 확인하지 못했어요`
  - `saving`: `저장 중`
  - `saved`: `저장 완료`
- 시작 전 `ready`, `checking`, `invalid`, `failed` 칩은 제거할 수 있다.
- 저장 시작 후 제거 동작은 제공하지 않는다. 진행 중 개별 취소는 후속 이슈 #8에서 다룬다.

### 오른쪽: 오늘 작업

- 저장 폴더를 폴더 아이콘과 함께 항상 표시한다.
- 폴더가 없으면 `저장할 폴더가 아직 없습니다`와 `저장 폴더 고르기`를 보여준다.
- 폴더가 있으면 `선택한 저장 폴더`와 폴더 이름을 보여준다.
- 브라우저 보안상 전체 경로는 표시하지 않고 폴더 이름만 표시한다.
- 진행 요약을 보여준다: `4대 중 3대 저장 완료`
- 완료 상태는 초록색 배지, 체크 아이콘, 굵은 텍스트로 구분한다.
- 데스크톱 알림은 작은 보조 버튼 `완료 알림 켜기`로 제공한다.

### 진행 목록

- 사용자에게 `truck-1`, URL 원문을 기본 표시하지 않는다.
- 매물명과 상태를 우선 표시한다.
- 예:
  - `[활어차] 포터2 슈퍼캡/초장축/(CRDi)` · `저장 완료`
  - `[카고] 마이티` · `저장 중 · 사진 12/18`
  - `주소 확인 필요` · `지우고 다시 붙여넣어 주세요`

## 데이터 모델

새 모델은 `PreparedListing` 개념을 둔다.

```ts
interface PreparedListing {
  id: string
  url: string
  status: 'checking' | 'ready' | 'invalid' | 'failed' | 'saving' | 'saved'
  label: string
  listing?: TruckListing
  message?: string
}
```

상태 의미:

- `checking`: URL을 찾았고 매물 정보를 확인 중이다.
- `ready`: 매물 정보를 확인했고 저장 시작 대상이다.
- `invalid`: 지원하지 않는 주소이거나 매물 URL을 찾지 못했다.
- `failed`: URL은 처리 대상이지만 매물 정보를 가져오지 못했다.
- `saving`: 저장 중이다.
- `saved`: 저장이 끝났다.

기존 batch store의 저장/진행 상태는 유지하되, 사용자에게 보이는 준비 목록과 진행 목록은 `PreparedListing`의 매물명 중심 데이터를 기준으로 구성한다.

## URL 추출

- 붙여넣은 텍스트에서 `truck-no1` 매물 상세 URL만 자동 추출한다.
- 줄바꿈, 공백, 주변 문장, 카톡/문자 문구가 섞여도 URL을 찾는다.
- 중복 URL은 새 칩을 만들지 않는다.
- 중복이 감지되면 기존 칩을 짧게 강조하거나 `이미 넣은 매물이에요`를 보여준다.
- 유효한 URL이 없으면 `매물 주소를 찾지 못했어요. 복사한 내용을 다시 확인해 주세요.`를 보여준다.

## 저장 시작 규칙

- 버튼은 상태에 따라 바뀐다.
  - 확인된 매물이 없고 확인 중도 없을 때: `매물 주소를 넣어주세요`
  - 확인 중인 매물이 있을 때: `매물 이름 확인 중`
  - 확인된 매물이 있을 때: `확인된 N대 저장 시작`
- 저장 대상은 `ready` 상태만 포함한다.
- `checking`이 남아 있으면 저장 버튼은 비활성화한다.
- `failed`와 `invalid`는 저장 대상에서 제외하고 화면에 남긴다.
- 저장 중에는 칩 제거를 막아 사용자가 취소로 오해하지 않게 한다.

## 오류와 복구

- 잘못된 주소: `매물 주소를 찾지 못했어요. 지우고 다시 붙여넣어 주세요.`
- 확인 실패: `매물 이름을 확인하지 못했어요. 잠시 후 다시 붙여넣어 주세요.`
- 중복 주소: `이미 넣은 매물이에요.`
- 저장 실패: 진행 목록에서 `저장하지 못했어요`와 `다시 시도`를 보여준다.
- 실패 칩은 URL 직접 편집을 제공하지 않는다.
- 복구 방식은 `지우기 -> 다시 붙여넣기`다.

## 알림

- 모달은 사용하지 않는다.
- 저장 완료는 진행 상황 영역 상단의 완료 요약으로 알린다.
- 데스크톱 알림은 선택형이다.
- 알림 권한을 거절해도 작업은 계속된다.
- 알림 문구 예: `트럭 매물 4대 저장이 끝났습니다.`

## 접근성

- 칩은 키보드로 초점을 받을 수 있어야 한다.
- 제거 버튼은 `매물 지우기: [매물명]`처럼 읽힌다.
- 상태 변화는 `aria-live="polite"` 영역으로 알린다.
- 오류 칩은 색만으로 구분하지 않고 텍스트와 아이콘을 함께 사용한다.
- 저장 시작 버튼은 비활성화 이유를 화면 텍스트로 설명한다.

## 테스트 계획

- 붙여넣은 문장 속에서 여러 URL을 추출하는 단위 테스트
- 공백, 줄바꿈, 카톡/문자 문구가 섞인 입력 테스트
- 중복 URL을 한 번만 칩으로 만드는 테스트
- `checking -> ready` 상태 전환 테스트
- `checking -> failed` 상태 전환 테스트
- 실패 칩에서 `지우기 -> 다시 붙여넣기` 흐름 테스트
- 시작 전 칩 제거 테스트
- 시작 후 제거 버튼이 보이지 않는 테스트
- 저장 버튼 문구와 비활성화 조건 테스트
- 진행 목록이 `truck-1` 대신 매물명을 표시하는 테스트
- 저장 완료 요약과 완료 배지 테스트
- 데스크톱 알림 권한 허용/거절 테스트
- E2E: 카톡처럼 섞인 텍스트를 붙여넣고 매물명 칩으로 바뀐 뒤 저장 완료까지 확인

## 범위 밖

- 진행 중 매물 개별 취소는 이번 범위에서 제외한다. 후속 이슈: https://github.com/jaem1n207/truck-harvester/issues/8
- 실패 칩 안에서 URL을 직접 수정하는 기능은 제외한다.
- legacy `/` route, `src/shared`, `src/widgets`, `src/app/page.tsx` 동작 변경은 제외한다.
- v2 watermark 기능 추가는 제외한다.
- v2 Sentry 추가는 제외한다.
