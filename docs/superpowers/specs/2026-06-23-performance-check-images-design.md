# 성능점검기록부 이미지 저장 설계

## 배경

Truck Harvester 사용자는 트럭 매물 주소를 붙여넣고, 네이버 스마트스토어
상품 등록에 필요한 차량 이미지와 원고 파일을 차량별 폴더로 저장한다.
실제 등록 결과의 상세 페이지에는 `성능기록지` 섹션이 있고, 이 섹션에는
성능점검기록부가 이미지 여러 장으로 등록된다.

현재 앱은 매물 페이지의 차량 이미지와 원고만 저장한다. 앞으로는 매물
페이지의 `성능점검보기` 링크를 따라 성능점검기록부도 함께 저장해야 한다.
성능점검기록부는 대부분 존재하지만, 드물게 없거나 열리지 않을 수 있다.

## 목표

- 매물 저장 시 차량 폴더 안에 성능점검기록부 JPG 이미지를 함께 저장한다.
- CheckPaper 원본 문서의 비율과 화질을 유지한다.
- 네이버 스마트스토어에 업로드하기 쉬운 파일명과 폴더 구조를 만든다.
- 성능점검기록부가 없거나 실패해도 차량 이미지와 원고 저장은 성공할 수
  있어야 한다.

## 비목표

- 성능점검기록부를 PDF로 저장하지 않는다.
- 성능점검기록부를 `848px` 같은 표시 폭으로 강제 축소하지 않는다.
- 원본 CheckPaper HTML을 사용자 산출물로 저장하지 않는다.
- 네이버 스마트스토어에 직접 업로드하는 자동화는 이번 범위에 포함하지
  않는다.

## 선택한 접근

CheckPaper 페이지의 각 A4 `.page` 요소를 고해상도 JPG로 캡처한다.

샘플 CheckPaper 문서는 `.page` 단위로 구성되어 있고 `(1/4쪽)`처럼 실제
문서 페이지 경계를 갖는다. 전체 긴 문서를 임의 높이로 자르면 표나 서명이
중간에서 끊길 수 있으므로, 원본 페이지 단위를 그대로 유지하는 방식이 가장
안정적이다.

캡처는 브라우저 렌더링 결과를 기반으로 한다. 문서 폭과 비율은 CheckPaper
HTML의 A4 레이아웃을 따르고, 캡처 scale을 높여 글자와 표 선이 스마트스토어
상세 이미지로 등록해도 읽히도록 한다. 결과 파일 포맷은 JPG를 사용한다.

## 저장 구조

기존 차량 폴더의 루트에 이미지와 원고가 바로 섞이지 않도록 하위 폴더를
만든다.

```text
저장 폴더/
  서울80바1234/
    차량 이미지/
      사진_1.jpg
      사진_2.jpg
    성능점검기록부/
      성능점검기록부_1.jpg
      성능점검기록부_2.jpg
    원고/
      차량정보.txt
```

파일명은 Windows 사용자를 기준으로 안전한 문자만 사용한다. 차량번호 폴더
이름은 기존처럼 위험 문자를 `_`로 바꾼다.

## 데이터 흐름

1. `POST /api/v2/parse-truck`가 매물 HTML을 가져온다.
2. Cheerio parser가 기존 차량 정보와 이미지 URL을 추출한다.
3. parser가 `성능점검보기` 링크를 찾는다.
   - 예시 매물은 `autocafe.co.kr/ASSO/CarCheck_Form_my.asp?OnCarNo=...`를
     가리키고, 이 URL이 최종 CheckPaper URL로 redirect 된다.
   - parser는 매물 페이지에 있는 링크를 절대 URL로 정규화해
     `performanceCheckUrl` optional field로 반환한다.
4. 저장 workflow가 차량 이미지와 원고를 저장한다.
5. `performanceCheckUrl`이 있으면 성능점검기록부 캡처 use case를 실행한다.
6. CheckPaper 페이지에서 `.page` 요소들을 찾고 페이지별 JPG bytes를 만든다.
7. 차량 폴더의 `성능점검기록부/` 아래에 `성능점검기록부_1.jpg`부터 저장한다.

## 성능점검기록부 캡처

캡처 use case는 `src/v2/features/file-management` 아래에 둔다. 저장 레이어가
이미 File System Access API와 ZIP fallback을 소유하고 있으므로, 성능점검기록부
이미지 생성과 파일 배치도 같은 feature 경계 안에서 다룬다.

캡처 구현은 사용자의 브라우저에서 실행한다. CheckPaper HTML은 외부 CSS,
이미지, 스크립트로 최종 문서를 채우므로 서버의 Cheerio 변환만으로는 정확한
결과를 얻기 어렵다. 다만 CheckPaper는 외부 도메인이므로, 브라우저가 직접
DOM을 읽거나 canvas로 내보낼 수 없도록 막힐 수 있다. 이를 피하기 위해
서버 API가 CheckPaper HTML과 이미지 asset을 same-origin으로 proxy하고,
클라이언트는 proxy된 HTML을 숨겨진 iframe에 렌더링한 뒤 `.page` 요소를
캡처한다.

DOM 캡처는 `html2canvas`를 사용한다. Playwright나 Puppeteer를 production 저장
경로에 추가하지 않는다. 성능점검기록부 저장은 Windows 사용자의 브라우저 저장
흐름 안에서 끝나야 한다.

세부 기준:

- 출력 포맷: `image/jpeg`
- 품질: `0.92` 수준의 높은 품질을 기본으로 하되 과도한 용량은 피한다.
- 크기: CheckPaper A4 비율을 유지한다.
- scale: 기본 `2`를 사용해 A4 문서 폭이 대략 1600px급으로 나오게 한다.
  `848px` 같은 표시 폭으로 축소하지 않는다.
- 페이지 단위: `.page` 하나가 JPG 하나가 된다.
- 실패 허용: `.page`가 없거나 캡처가 실패하면 성능점검기록부만 건너뛴다.

## 저장 진행률

현재 진행률은 사진 개수만 기준으로 표시한다. 이번 변경에서 사용자에게
기술적인 문서 처리 단계를 노출하지 않는다.

저장 중에는 기존처럼 사진 저장 진행률을 유지하고, 성능점검기록부는 같은
차량 저장 작업의 후속 단계로 처리한다. 성능점검기록부 캡처 실패는 해당 차량
전체 저장 실패로 바꾸지 않는다.

이번 범위에서는 성능점검기록부 누락 여부를 별도 상태로 표시하지 않는다.
성능점검기록부 저장 여부를 명시해야 한다는 요구가 생기면, 별도 변경으로
상태 모델에 `성능점검기록부 저장됨/없음/실패` 같은 보조 표시를 추가한다.

## ZIP fallback

폴더 저장을 지원하지 않는 브라우저에서는 ZIP 안에도 같은 구조를 만든다.

```text
서울80바1234/
  차량 이미지/
  성능점검기록부/
  원고/
```

성능점검기록부 캡처가 브라우저 렌더링에 의존하므로 ZIP 생성 전에도 동일한
JPG bytes를 만든 뒤 ZIP folder에 넣는다. 캡처가 실패하면 ZIP에는
성능점검기록부 폴더를 만들지 않는다.

## 에러 처리

- 매물 HTML에 `성능점검보기` 링크가 없으면 성능점검기록부 저장을 생략한다.
- 최종 CheckPaper URL이 열리지 않으면 성능점검기록부 저장을 생략한다.
- CheckPaper 페이지에 `.page` 요소가 없으면 성능점검기록부 저장을 생략한다.
- 차량 이미지와 원고 저장이 성공했다면 차량 저장은 성공으로 본다.
- 성능점검기록부 생략은 사용자에게 별도 실패 문구로 노출하지 않는다.

## 테스트 계획

- parser unit test
  - `성능점검보기` 링크를 `performanceCheckUrl`로 추출한다.
  - 링크가 없으면 field가 undefined로 남는다.
- entity schema test
  - `performanceCheckUrl` optional URL field를 허용한다.
- file naming test
  - `사진_1.jpg`, `성능점검기록부_1.jpg`, `차량정보.txt` 이름을 안정적으로
    만든다.
- file-system test
  - 차량 이미지, 성능점검기록부, 원고가 새 폴더 구조로 저장된다.
  - 성능점검기록부 캡처 실패가 차량 저장 실패로 전파되지 않는다.
- zip fallback test
  - ZIP 안에 동일한 폴더 구조와 JPG 파일이 들어간다.
- workflow test
  - `performanceCheckUrl`이 있는 listing은 캡처 use case를 호출한다.
  - `performanceCheckUrl`이 없는 listing은 기존 저장 흐름을 유지한다.

## 구현 경계

새 서버 API는 CheckPaper HTML fetch, redirect 추적, asset proxy만 담당한다.
JPG 생성은 클라이언트 캡처 use case가 담당한다. 이 경계를 유지하면 서버
runtime에 무거운 headless browser를 올리지 않고도, 사용자가 실제로 보는
문서 비율과 렌더링 결과를 저장할 수 있다.
